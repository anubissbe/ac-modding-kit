# Compatibility policy

This policy applies to Ancient Cities Modding Kit releases starting with `0.2.0a1`.
It separates toolkit compatibility from game compatibility so that a toolkit release
never implies support for every Ancient Cities build.

## Compatibility dimensions

Every compatibility claim has three independent parts:

1. **Toolkit version** uses PEP 440 version strings and follows Semantic Versioning for
   the documented public Python API and command-line interface.
2. **Python version** is the interpreter range declared by the release metadata and
   exercised by CI. The current range is Python 3.11 through 3.14.
3. **Game build** is identified by the Ancient Cities semantic version, Steam build ID,
   internal `GameVersion`, and, when available, a content hash. A toolkit release may
   understand several game builds, but runtime compatibility is only verified against
   the exact build tuple recorded by the test evidence.

Static validation means that a project satisfies the checks implemented by the
toolkit. It does not prove that the game will load the mod or that a save, achievements,
or other mods will remain unaffected.

## Release stability

Versions below `1.0.0` are previews. Alpha releases may change before the corresponding
final minor release. Outside an explicitly marked alpha or beta series, incompatible
changes to a documented public interface require a new minor `0.x` release and a
migration note.

Starting with `1.0.0`, releases follow Semantic Versioning:

- patch releases contain compatible fixes;
- minor releases add compatible functionality; and
- major releases may remove or incompatibly change a public contract.

Security, legal, or data-loss risks may require an earlier incompatible change. Such a
change must be called out prominently in the changelog and release notes.

## Public contracts

The compatibility policy covers:

- names documented as public from the `acmk` package;
- the `acmk` console command, documented subcommands and options, exit-code meanings,
  and machine-readable output fields;
- invocation through `python -m acmk`;
- the legacy `ancient_cities_mod` import and script entry point while it is documented
  as supported;
- Markdown knowledge resources exposed through `acmk.knowledge_data`; and
- versioned JSON schemas exposed through `acmk.schemas`.

Modules, names, and attributes beginning with an underscore are internal. Test helpers,
undocumented implementation modules, human-readable prose, and exact diagnostic wording
are not stable contracts.

For new SDK commands, `--json` emits the report-envelope schema; default pretty output is
the unwrapped command payload. Compatible releases may add optional command-specific fields
inside the envelope's open `data` object. The closed envelope root changes only with a report
schema version change. Consumers must ignore unknown fields inside `data`. Removing a field,
changing its type or meaning, or changing an established exit-code meaning is incompatible.

## Deprecation and removal

A public contract is deprecated before removal whenever safety permits. A deprecation
notice identifies the replacement and the earliest release that may remove the old
behavior. Deprecated behavior remains available for at least two feature releases.
After `1.0.0`, removal also requires a major release.

Raising the minimum Python version, removing a supported invocation style, or dropping a
documented game-build profile follows the same notice period. An upstream Python
end-of-life event or an urgent security issue may shorten it; the exception is documented
in the release notes.

## Game compatibility evidence

Claims that a feature works in game must record the exact game-build tuple, operating
system, toolkit version, test date, clean-launch status, save type, and relevant sanitized
log result. A static parser or validator test alone is not in-game evidence. New or
unknown builds remain unverified until this evidence is collected, even if their files
pass static validation.

The installed game, a skeleton generated by that build, and its runtime log take
precedence over older examples or community documentation. Compatibility fixes should
preserve older verified profiles where practical; otherwise the dropped profile follows
the deprecation policy above.

## Release checks

A releasable change must pass formatting and lint checks, strict type checking for
`src/acmk`, the supported Python test matrix, and source compilation. Built wheels must
be installed into an isolated environment and successfully import `acmk`,
`acmk.knowledge_data`, `acmk.schemas`, and `ancient_cities_mod`; both `acmk` and `python -m acmk` must
start successfully. Distribution checks also verify that the packaged knowledge Markdown
is readable through Python's resource API.
